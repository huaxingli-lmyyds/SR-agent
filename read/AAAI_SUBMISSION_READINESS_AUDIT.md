# SR-Agent 代码审查与 AAAI 投稿准备度报告

> 审查日期：2026-06-18  
> 审查范围：`SR-agent` 项目，明确排除 `speechbrain/` 仓库镜像代码。  
> 证据边界：本报告依据当前源代码、配置和测试。没有论文初稿、完整训练日志、GPU 复现实验结果或外部基线结果，因此不对算法效果和录用概率作结论。

## 1. 结论摘要

项目已形成一个结构化的语音研究智能体原型：数据处理智能体负责数据检查、处理计划、版本发布和下游交接；超参数智能体负责搜索空间、策略选择、多保真调度、失败重试、反馈调整、跨 Study campaign 和实验追踪。工程工作量已经明显超过演示脚本，但当前证据主要证明“软件结构可运行”，尚未证明“方法在科学上新颖且显著优于强基线”。

当前判断：

- 工程原型成熟度：中等偏上。
- 接口与可审计性：中等偏上，已有结构化请求/响应、策略决策记录、数据 lineage 和实验 tracker。
- 实际端到端验证：不足；当前环境缺少部分直接依赖，LangGraph、工具边界和 Optuna 路径未执行。
- 科学贡献与实验充分性：目前不可评估，尚无论文 claim、基线对照、消融、统计显著性和跨数据集证据。
- AAAI 主会准备度：暂不建议直接投稿；应先把项目从“智能体编排系统”收敛成一个可证伪的研究问题并完成系统实验。

## 3. 超参数智能体审查

### 3.1 已具备的功能

- 统一入口：`AgentTaskRequest -> HPOAgent -> AgentTaskResult`。
- 搜索策略：随机搜索、网格搜索、自适应搜索、可选 Optuna TPE、Successive Halving。
- 计划控制：结构化搜索空间、目标、预算 rung、最大训练次数和策略 proposal/decision 审计。
- 执行控制：LangGraph scheduler、trial 状态机、失败分类、有限重试、晋级和严格完成检查。
- 在线反馈：基于完成 trial、失败聚类和边界命中的策略复审，仅影响后续候选生成。
- Campaign：支持多个 Study、目标值、patience、最小改进和总训练次数上限。
- 工程解耦：task/model/runner registry 和 SpeechBrain runner 适配器。
- 可追踪性：Study、Trial、指标、参数、产物、策略决策和 campaign 均写入实验记录。

### 3.2 本次发现并修复的问题

1. **默认搜索参数被预算静默覆盖**  
   默认搜索空间包含 `number_of_epochs`，但训练工具会用 `budget.epochs` 覆盖它，导致该维度看似被优化、实际不生效。现已从默认搜索空间删除，并在用户显式同时配置二者时抛出清晰错误。

2. **LLM 接口依赖声明不完整**  
   源码直接使用 `langchain-core`、`langchain-openai` 和 `python-dotenv`，但项目依赖未直接声明。现已补入 `pyproject.toml`。

3. **智谱 API Base 环境变量拼写不兼容**  
   原代码仅读取疑似拼写错误的 `ZHIU_API_BASE_URL`。现优先读取 `ZHIPUAI_API_BASE_URL`，并保留旧名称兼容。

### 3.3 仍需重点验证的风险

- 真实 GPU 训练、超时子进程、checkpoint 收集、评估和失败重试尚未在本次环境中端到端执行。
- `TrainModel` 与 scheduler 都会更新 trial 生命周期，虽然当前状态机容许重复状态写入，但长期建议收敛为单一状态所有者，避免并发或异常路径产生不一致。
- 目前主要是单目标优化；若论文主张“资源感知”，应显式支持并评估准确率、时间、显存、能耗或成本的多目标 Pareto 优化。
- 自适应策略的理论依据和与成熟 HPO 方法的区别尚未由代码证据建立。
- LLM advisor 默认关闭；若论文贡献依赖 LLM，必须证明它相对确定性 policy 的增益、成本和稳定性。

## 4. 数据处理智能体审查

### 4.1 已具备的功能

- 数据集类型推断与文件/CSV manifest profile。
- 缺失源、空数据、空文件、同名文件、manifest 空值等质量问题检测。
- 注册式 processor 接口和参数校验。
- 确定性计划构建、执行后规则验证、失败短路。
- 默认内置 `validate_dataset` 与 `filter_manifest_rows`。
- 非覆盖式派生数据、数据版本、parent version、artifacts 和消费状态。
- `data_handoff` 明确区分 `ready`、`source_unchanged` 与 `not_ready`，阻止 HPO 静默使用不完整派生数据。
- LLM 仅能提出受约束建议，不能启用受保护的大规模 materialization 参数。

### 4.2 本次发现并修复的问题

1. **工具接口不能传递显式处理操作**  
   `BuildDataProcessingPlan` 原先只能根据 profile 自动建计划，无法通过工具接口传入 processor 与参数。现新增 `requested_operations_json`，要求为 JSON object 数组并进行类型/processor 校验。

2. **派生输出可写入源数据目录内部**  
   非 materialize 模式原先允许输出目录成为源目录子目录，会改变源数据树并可能被后续扫描再次读入。现一律拒绝源目录内部输出。

3. **递归 CSV 同名覆盖**  
   使用 `**/*.csv` 时，不同子目录的同名 manifest 原先会写到同一个目标文件。现保留相对目录结构并同步修正 artifact 名称。

4. **扫描上限缺少参数边界**  
   `max_files <= 0` 原先仍可能扫描一个文件。现要求严格正整数。

### 4.3 仍需重点优化的风险

- 内置处理能力目前很窄，主要是 manifest 清洗与验证，尚不足以支撑“通用数据处理智能体”的强 claim。
- 音频领域缺少采样率/声道/时长/削波/静音/SNR/损坏音频检查、说话人泄漏检测、重复音频检测、数据分布漂移和 train/dev/test 交叉污染检查。
- 默认 manifest 清洗产生的只是派生 manifest；若未安全 materialize 完整数据，版本会正确标记为 `not_ready`。论文实验需明确采用哪种可消费数据集构建策略，避免复制完整 VoxCeleb 的高存储成本。
- profile 采用扫描上限，当前没有随机/分层采样及置信区间；大规模数据上的 profile 可能有偏。
- 本地路径支持较完整，但远程对象存储、dataset hub 和事务性发布尚未形成统一实现。

## 6. AAAI 研究定位建议

不要把论文主张停留在“我们把两个 agent 接起来”。更可行的中心问题是：

> 在严格训练预算和数据质量不确定性下，受约束、可审计的多智能体决策是否能比固定数据流水线与标准 HPO 更高效、更稳健地获得更好的语音验证模型？

建议只选一个主贡献轴，另一个作为系统支撑：

- **优先方向 A：闭环数据质量—HPO 联合优化。** 数据智能体发现质量问题并发布可验证版本，HPO 智能体依据数据 lineage 与失败反馈调整搜索；重点证明联合闭环优于两阶段固定流水线。
- **方向 B：受约束 LLM advisor。** 研究结构化 proposal、确定性 validator 和可审计执行器如何降低自由代理的无效/危险动作；重点评估有效建议率、违规拦截率、性能增益与 token/时间成本。
- **方向 C：预算感知的鲁棒 HPO。** 以失败分类、动态策略切换和 campaign 为核心；需要与 ASHA/Hyperband、Optuna TPE、BOHB 等强基线比较。

当前代码尚不能支持“通用 AutoML”或“通用数据智能体”的宽泛结论，除非扩展到多个任务、数据集和模型族。

## 7. 必须增加的实验工作量

### 7.1 最小可投稿实验矩阵

| 维度 | 最低建议 |
|---|---|
| 数据集 | VoxCeleb1 + VoxCeleb2；最好增加 CN-Celeb 或另一个域外说话人数据集 |
| 模型 | ECAPA-TDNN + 至少一个不同归纳偏置的模型（ResNet/x-vector/SSL downstream） |
| 固定基线 | 默认人工配置、随机搜索、网格搜索（小空间） |
| 强 HPO 基线 | Optuna TPE、ASHA/Hyperband；若主张贝叶斯优势，再加入 BOHB/SMAC 类基线 |
| Agent 基线 | 无 LLM 的确定性 policy、自由文本 LLM agent、当前受约束 advisor |
| 数据基线 | 不清洗、固定规则清洗、仅数据智能体、仅 HPO、联合闭环 |
| 重复实验 | 每个关键设置至少 3–5 个随机种子 |
| 任务指标 | EER、minDCF；另报 wall-clock、GPU-hours、失败率、峰值显存和实验成本 |
| 鲁棒性 | 数据缺失、标签/manifest 噪声、损坏音频、域偏移、预算收紧 |

### 7.2 关键消融

1. 去掉数据 handoff/lineage，只传路径。
2. 去掉反馈分析与运行时策略复审。
3. 去掉 campaign，只运行单 Study。
4. 去掉 validator，让 LLM 直接给动作，统计失败与违规。
5. LLM advisor 关闭、不同模型、不同温度和重复调用稳定性。
6. 固定搜索空间与可调整搜索空间。
7. 固定预算、Successive Halving、动态预算。
8. 不同数据 profile 扫描率对决策质量与耗时的影响。

### 7.3 统计与报告规范

- 所有主结果报告均值、标准差或置信区间，并使用配对检验或 bootstrap 比较关键方法。
- 预先定义主指标、预算和停止规则，避免事后挑选最佳实验。
- 同时报告 best-found performance 与 anytime curve（性能—GPU hours）。
- 记录失败 trial，而不是从统计中删除；失败率本身应作为系统可靠性指标。
- 发布完整配置、随机种子、数据版本 hash、环境锁文件和运行命令。

## 10. 三位审稿侧重点模拟

### Reviewer 1：技术可靠性

- **总体评价**：系统分层、契约、审计和测试基础不错，但真实端到端证据不足。
- **主要优点**：结构化 proposal、确定性验证、明确状态机、数据版本交接和失败记录。
- **主要问题**：依赖环境未完整，核心 LangGraph/TPE 路径被跳过；没有真实 GPU campaign、异常恢复和多种子结果。
- **建立论文结论前必须解决**：完整复现、强基线、公平预算、统计检验、失败率与资源成本。
- **倾向**：工程潜力明确，但以当前证据无法支持技术有效性结论。

### Reviewer 2：原创性与重要性

- **总体评价**：组件组合本身可能被视为增量式系统工程，原创性取决于闭环决策机制是否带来可重复优势。
- **主要优点**：数据质量、HPO 和受约束 agent 的交叉点有研究空间。
- **主要问题**：尚未明确与标准 AutoML、workflow orchestration、LLM-agent guardrail 和现有 HPO 的方法区别。
- **建立论文结论前必须解决**：凝练一个中心假设，给出方法形式化描述，并通过强基线与消融证明每个新机制的独立贡献。
- **倾向**：若只有系统实现，主会说服力偏弱；若闭环方法在多数据集和固定预算下稳定领先，则有机会形成贡献。

### Reviewer 3：广泛兴趣与可读性

- **总体评价**：可审计、预算受控的研究智能体具有跨领域吸引力，但当前实现与 VoxCeleb/ECAPA/SpeechBrain 绑定仍较明显。
- **主要优点**：问题容易被 AutoML、LLM agent、语音和可复现研究社区理解。
- **主要问题**：缺少非语音或至少第二任务证据；项目文档与包装不完整，外部读者难以复现。
- **建立论文结论前必须解决**：清晰任务定义、方法图、执行示例、匿名 artifact、跨模型/数据集验证和失败案例分析。
- **倾向**：可读性和影响力取决于是否把系统细节抽象成一般机制，而不是把工程接口逐项写成贡献。

## 11. Cross-review synthesis

- 共识优势：工程基础、审计边界、数据 lineage、预算/失败控制和可扩展 registry。
- 共识风险：科学 claim 未定义，核心路径未完整运行，缺少强基线、多数据集、多种子、统计与资源成本。
- 权重差异：Reviewer 1 更关注可靠性，Reviewer 2 更关注方法新颖性，Reviewer 3 更关注泛化与复现。
- 最优先事项：先完成 P0；随后选择一个中心研究方向，做小规模预实验验证信号，再投入完整矩阵。
- 不支持的结论：目前不能声称优于现有 HPO、降低训练成本、提高 EER/minDCF、具备通用性或达到 AAAI 录用标准。
